# Skill Finder

在 GitHub 上搜索符合用户需求的高质量开源项目（不限于特定生态，覆盖工具、库、框架、CLI、Skill 等各类项目），按量化指标筛选，返回经过验证的、稳定的、广受好评的结果。

## 触发条件

当用户提出以下意图时触发：
- "帮我找一个 XXX 的工具/库/skill"
- "搜索 GitHub 上有没有 XXX 相关的项目"
- "推荐一个能做 XXX 的开源项目"
- "有没有好用的 XXX"
- 任何涉及搜索、发现、推荐开源项目/Skill 的请求

## 搜索策略

### 1. 解析用户需求

从用户输入中提取：
- **核心功能**：用户想要什么能力
- **领域关键词**：技术栈、平台、场景等
- **偏好约束**：语言、复杂度、使用方式等

### 2. 构建搜索查询

使用通用 GitHub 搜索模式，按优先级排列：

```
# 主查询：按功能关键词 + 质量信号搜索
<keywords> stars:>50 pushed:>2025-12-28 site:github.com

# 补充查询：按 awesome 列表搜索（精选合集质量更高）
awesome <keywords> site:github.com

# 补充查询：按 GitHub topic 搜索
topic:<topic-keyword> <keywords> site:github.com
```

使用 WebSearch 执行搜索，每次查询最多返回 10 条结果。至少执行 2 轮不同关键词组合的搜索。

### 3. 量化质量筛选

对每个候选仓库，必须通过以下标准评估。**不满足任一核心标准的直接排除**。

#### 核心标准（必须全部满足，否则排除）

| 指标 | 阈值 | 说明 |
|------|------|------|
| GitHub Stars | >= 50 | 太少说明缺乏社区验证 |
| 最近更新 | 6 个月内（2025-12-28 之后） | 超过 6 个月未更新视为不活跃 |
| 有 README | 必须有 | 无文档说明质量不可靠 |

#### 加分标准（满足越多越好，用于排序）

| 指标 | 阈值 | 说明 |
|------|------|------|
| Stars | >= 500 | 高度认可 |
| Open Issues 比例 | < 15% | Issues 少说明维护良好 |
| 最近 release | 3 个月内 | 有正式 release 更稳定 |
| Contributors | >= 5 | 多人维护更可持续 |
| 有测试 | 存在 test/ 目录或 CI 配置 | 有测试说明质量意识强 |
| License | 有开源许可证 | 确保可合法使用 |
| 在 awesome 列表中 | 收录于 awesome-* 合集 | 被社区精选收录 |

### 4. 信息获取流程

对每个候选仓库：
1. 先用 WebSearch 获取仓库标题、描述、stars、语言、最后更新时间
2. 对符合核心标准的仓库，用 WebFetch 访问其 GitHub 页面获取详细信息：
   - 完整 README
   - Open/Closed Issues 数量
   - Contributors 数量
   - 最近 Release 信息
3. 必要时访问项目官网或文档站确认更多细节

### 5. 结果输出格式

将所有通过筛选的项目按以下格式输出：

```markdown
## 搜索结果：<用户需求摘要>

### 推荐排序

| # | 项目名称 | Stars | 最近更新 | 评分 | 简介 |
|---|---------|-------|----------|------|------|
| 1 | owner/repo | 1.2k | 2026-05 | ★★★★★ | 一句话描述 |

### 详细评估

#### 1. [项目名称](GitHub URL)

- **Stars**: xxx  |  **最近更新**: YYYY-MM-DD  |  **License**: MIT
- **简介**: 完整功能描述
- **核心能力**: 
  - 能力点 1
  - 能力点 2
- **质量评估**:
  - Open Issues 比例: xx%  ✅
  - Contributors: xx 人  ✅
  - 有测试: 是/否
  - 有 Release: 是/否
- **使用方式**: 安装/使用方法简述
- **推荐理由**: 为什么适合用户需求

---

### 总结

共找到 X 个符合条件的项目，推荐 TOP 3 优先尝试。
```

## 重要约束

1. **必须验证**：每个推荐的项目必须经过 WebFetch 访问 GitHub 页面确认信息真实性，不得仅凭搜索摘要推荐
2. **宁缺毋滥**：如果没有满足核心标准的项目，诚实告知用户，不要推荐不合格的
3. **最多返回 5 个**：只推荐评分最高的前 5 个，避免信息过载
4. **必须附链接**：每个推荐必须附带可点击的 GitHub URL
5. **使用中文输出**：所有结果、评估、说明均使用中文
6. **不限于特定生态**：搜索范围覆盖所有 GitHub 开源项目，不限定 TRAE、VS Code、npm 等特定平台
